# multicast_health_check.py 组播地址健康检查工具

## 功能概述

用于测试组播地址（rtp/udp）的可用性，通过 udpxy 服务器代理访问。支持多种输入格式，自动识别地区和运营商，生成多个输出文件。

## 处理流程

### 1. 文件选择与地区识别

```
弹出文件选择对话框（默认目录: multicast_addresses\collected）
    ↓
选择单个.txt文件
    ↓
从文件名或目录名提取地区和运营商（如"广东电信.txt"提取"广东"和"电信"）
    ↓
├─ 能提取到地区 ──→ 使用提取的地区选择服务器
└─ 无法提取 ──→ 显示提示信息并退出
```

### 2. 解析文件内容

```
逐行读取文件（支持多种编码：utf-8, gbk, gb2312, latin1）
    ↓
识别行类型：
├── #genre# 或分类名,#genre# ──→ 分类标记（current_category = 分类名）
├── 分类,频道名,地址,无效 ──→ 四段式格式（含标记）
├── 分类,频道名,地址 ──→ 三段式格式
├── 频道名,地址 ──→ 两段式格式
└── 纯地址（rtp://... 或 udp://... 或 IP:Port） ──→ 直接使用
```

### 3. 测试配置

```
命令行参数：
├── -f, --file: 指定组播地址文件路径
├── -s, --servers: 最大服务器数量（默认 30）
└── -a, --all: 是否强制测试所有地址（包括标记为无效的）

读取 nodes.txt 中的 udpxy 服务器列表（按地区分组）
根据文件名提取的地区选择对应的服务器
```

### 4. 测试执行

```
收集所有分类的地址统一测试
    ↓
按服务器数量分组（每个服务器测试约 N/M 个地址）
    ↓
并发测试（ThreadPoolExecutor，并发数=服务器数）
    ↓
调用 test_multicast_url(addr, server)
    ↓
├─ 下载≥16KB ──→ [√] 有效 → valid_results
└─ 下载<16KB ──→ [×] 无效 → active_invalid
```

### 5. 结果保存

```
生成三个输出文件：
├─ 1. {源文件名}_health_checked.txt: 完整测试结果（含有效/无效标记）
├─ 2. {地区}_{运营商}_valid_multicast.txt: 有效组播地址（去重、追加模式）
└─ 3. {地区}{运营商}.txt: 复制到 source/multicast/ 目录
```

## 文件格式

### 支持的输入格式

```
#genre# 央视频道                    ← 分类标记
CCTV1,http://udpxy_server/rtp/239.49.0.1:8000   ← 带 udpxy 服务器的地址
央视,CCTV-1,239.49.0.1:8000   ← 三段式格式：分类,频道名,组播地址（无 rtp:// 前缀）
央视,CCTV-2,rtp://239.49.0.2:8000   ← 带 rtp:// 前缀的地址
湖南卫视,rtp://239.49.0.3:8000   ← 两段式格式：频道名,组播地址
rtp://239.49.0.4:8000   ← 纯地址格式
239.49.0.5:8000   ← 纯IP地址格式（自动添加 rtp:// 前缀）
```

#### 1. health_checked.txt（完整测试结果）

```

# 格式: 分类,频道名,组播地址,状态
#
央视,CCTV-1,rtp://239.49.0.1:8000,有效
央视,CCTV-2,rtp://239.49.0.2:8000,有效
央视,CCTV-3,rtp://239.49.0.3:8000,无效
```

```
央视,CCTV-2,rtp://239.49.0.2:8000
```

## 变量说明

| 变量               | 说明                               |
| ---------------- | -------------------------------- |
| `items`          | {分类: {active: [], failed: []}} |
| `active`         | 当前有效的地址列表                        |
| `failed`         | 之前测试失败的地址列表                      |
| `valid_results`  | 本次测试有效的地址                        |
| `active_invalid` | 本次测试失败的活动地址                      |
| `valid_failed`   | 本次测试有效的失败地址（恢复）                  |
| `invalid_failed` | 本次测试仍然失败的地址                      |

## 使用方法

```bash
py multicast_health_check.py
```

### 命令行参数

```bash
# 基本使用
py multicast_health_check.py

# 指定文件
py multicast_health_check.py -f multicast_addresses/collected/江苏电信.txt

# 指定最大服务器数量
py multicast_health_check.py -s 50

# 强制测试所有地址（包括标记为无效的）
py multicast_health_check.py -a
```

## 输出文件

| 输出文件 | 路径 | 说明 |
|---------|-----|-----|
| `{源文件名}_{时间戳}.log` | 源文件所在目录 | 详细测试日志 |
| `{源文件名}_health_checked.txt` | 源文件所在目录 | 完整测试结果（含有效/无效标记） |
| `{地区}_{运营商}_valid_multicast.txt` | multicast_addresses/curated/ | 有效组播地址（去重、追加） |

## 标记说明

| 标记        | 说明                |
| --------- | ----------------- |
| `#genre#` | M3U playlist 分类标记 |
| `分类,频道名,地址,无效` | 四段式格式（含无效标记） |
| `分类,频道名,地址` | 三段式格式 |
| `频道名,地址`  | 两段式格式 |
| `rtp://IP:Port` 或 `udp://IP:Port` | 带协议前缀的纯地址格式 |
| `IP:Port`  | 不带协议前缀的纯IP地址格式（自动添加 rtp://） |


## 测试逻辑

`test_multicast_url(addr, server)` 函数：

1. 从地址中提取协议（rtp/udp）、IP、端口
2. 构建 udpxy URL: `http://server/protocol/ip:port`
3. 使用 requests 访问，设置超时
4. 检查是否下载了至少 16KB 数据
5. 返回：(原始地址, 是否有效, 状态信息, 测试URL)

## 注意事项

1. **编码问题**：自动尝试多种编码读取文件（utf-8 → gbk → gb2312 → latin1）
2. **地区识别**：优先从文件名识别，其次从目录名识别
3. **去重逻辑**：curated 文件采用追加模式，按组播地址去重，保留原始顺序
4. **分类优先级**："未分类"优先级最低，不会覆盖已有的分类信息
5. **服务器选择**：只使用与地区匹配的 udpxy 服务器进行测试

## 分类更新优先级

| 原有分类 | 新读取分类 | 处理方式 |
|---------|-----------|---------|
| 非"未分类" | "未分类" | 不更新 |
| "未分类" | 非"未分类" | 更新 |
| 非"未分类" | 非"未分类" | 更新为新值 |
| "未分类" | "未分类" | 不更新 |
